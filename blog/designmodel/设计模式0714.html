设计模式详解

1 设计模式

1.1 概念

设计模式是软件设计中常见问题的典型解决方案。 它们就像能根据需求进行调整的预制蓝图， 可用于解决代码中反复出现的设计问题。

设计模式与方法或库的使用方式不同， 你很难直接在自己的程序中套用某个设计模式。 模式并不是一段特定的代码， 而是解决特定问题的一般性概念。 你可以根据模式来实现符合自己程序实际所需的解决方案。

人们常常会混淆模式和算法， 因为两者在概念上都是已知特定问题的典型解决方案。 但算法总是明确定义达成特定目标所需的一系列步骤， 而模式则是对解决方案的更高层次描述。 同一模式在两个不同程序中的实现代码可能会不一样。

算法更像是菜谱： 提供达成目标的明确步骤。 而模式更像是蓝图： 你可以看到最终的结果和模式的功能， 但需要自己确定实现步骤。

1.2 设计模式原则

1.3 设计模式内容

大部分模式都有正规的描述方式， 以便在不同情况下使用。 模式的描述通常会包括以下部分：

  • 意图部分简单描述问题和解决方案。
  • 动机部分将进一步解释问题并说明模式会如何提供解决方案。
  • 结构部分展示模式的每个部分和它们之间的关系。
  • 在不同语言中的实现提供流行编程语言的代码， 让读者更好地理解模式背后的思想。

部分模式介绍中还列出其他的一些实用细节， 例如模式的适用性、 实现步骤以及与其他模式的关系。

2 常用设计模式

2.1 创建型模式

顾名思义，主要解决如何灵活创建对象或者类的问题。

2.1.1 创建造者模式

将一个复杂对象的构建与其表示分离，使得同样的构建过程可以创建不同的表示。

应用场景：

当一个类的构造函数参数个数超过4个，而且这些参数有些是可选的参数，考虑使用构造者模式。

举例说明：

当一个类的构造函数参数超过4个，而且这些参数有些是可选的时，我们通常有两种办法来构建它的对象。 例如我们现在有如下一个类计算机类Computer，其中cpu与ram是必填参数，而其他3个是可选参数，那么我们如何构造这个类的实例呢

public class Computer {
    private String cpu;//必须
    private String ram;//必须
    private int usbCount;//可选
    private String keyboard;//可选
    private String display;//可选
}

第一：折叠构造函数模式（telescoping constructor pattern ），这个我们经常用,如下代码所示：

xxxxxxxxxx
public class Computer {
     ...
    public Computer(String cpu, String ram) {
        this(cpu, ram, 0);
    }
    public Computer(String cpu, String ram, int usbCount) {
        this(cpu, ram, usbCount, "罗技键盘");
    }
    public Computer(String cpu, String ram, int usbCount, String keyboard) {
        this(cpu, ram, usbCount, keyboard, "三星显示器");
    }
    public Computer(String cpu, String ram, int usbCount, String keyboard, String display) {
        this.cpu = cpu;
        this.ram = ram;
        this.usbCount = usbCount;
        this.keyboard = keyboard;
        this.display = display;
    }
}

可以看出这种方式，太复杂，冗余！！！

那么我们采用建造者模式吧！

步骤：

  1. 在Computer 中创建一个静态内部类 Builder，然后将Computer 中的参数都复制到Builder类中。
  2. 在Computer中创建一个private的构造函数，参数为Builder类型
  3. 在Builder中创建一个public的构造函数，参数为Computer中必填的那些参数，cpu 和ram。
  4. 在Builder中创建设置函数，对Computer中那些可选参数进行赋值，返回值为Builder类型的实例
  5. 在Builder中创建一个build()方法，在其中构建Computer的实例并返回

代码如下：

​x
public class Computer {
    private final String cpu;//必须
    private final String ram;//必须
    private final int usbCount;//可选
    private final String keyboard;//可选
    private final String display;//可选


    private Computer(Builder builder){
        this.cpu=builder.cpu;
        this.ram=builder.ram;
        this.usbCount=builder.usbCount;
        this.keyboard=builder.keyboard;
        this.display=builder.display;
    }
    public static class Builder{ // 内部静态类
        private String cpu;//必须
        private String ram;//必须
        private int usbCount;//可选
        private String keyboard;//可选
        private String display;//可选


        public Builder(String cup,String ram){
            this.cpu=cup;
            this.ram=ram;
        }


        public Builder setUsbCount(int usbCount) {
            this.usbCount = usbCount;
            return this;
        }
        public Builder setKeyboard(String keyboard) {
            this.keyboard = keyboard;
            return this;
        }
        public Builder setDisplay(String display) {
            this.display = display;
            return this;
        }        
        public Computer build(){
            return new Computer(this);
        }
    }
  //省略getter方法
}

那么怎么使用它创建对象？这样的方式就可以非常灵活的构建对象了。

代码如下：

xxxxxxxxxx
Computer computer=new Computer.Builder("因特尔","三星")
                .setDisplay("三星24寸")
                .setKeyboard("罗技")
                .setUsbCount(2)
                .build();

总结：

实例化对象方式，需要编写不同的构造函数,在成员变量非常多的情况下需要根据可变的成员变量来编写大量不同的构造函数，灵活性较差，参数类型相似时实际调用时不方便。

建造者模式的优点 (1)建造者模式将类的构造函数中的可选参数分离出来，使用setter的方式进行初始化，非常的灵活。 (2)建造者模式是链式调用，属性连续设置，当调用build()方法实例化对象后对象不可再进行改变，看起来是一个整体，较简洁，易于程序员阅读和管理。

为什么建造者模式创建出的对象不可变？ 由于Computer的构造函数私有，使用build方法构造对象后，对象不可变。 为什么会有new Computer.Builder这样的写法？ 内部静态类的实例化模式如此：new Computer.Builder(参数1,参数2...) （类名.内部静态类名）

Computer computer=new Computer.Builder("因特尔","三星") .setDisplay("三星24寸") .setKeyboard("罗技") .setUsbCount(2) .build();

能不能把new 给去掉呀？为什么？

不可以。因为Builder 是Computer的内部类，那些set方法都是Builder类的实例方法，那个new是在创建Builder类的对象，有了对象你才可以调用其实例方法。computer对象是由builder类的实例方法build()创建的。

2.1.2 简单工厂模式

创建型设计模式讨论的是如何创建对象的问题。就是以各种高逼格的方式最终将一个类的对象new出来，一点也不神秘，简单工厂模式就属于此类型。

简单工厂模式，又叫做静态工厂方法（Static Factory Method）模式，是由一个工厂对象决定创建出哪一种产品类的实例。

应用场景：

首先当然是在你需要new一个类的对象的时候，此时各种状况出现啦：

  1. 你不想直接new这个类的对象，怕以后这个类改变的时候你需要回来改代码，而此时依赖这个类的地方已经到处都是了。
  2. 这个类的对象构建过程非常复杂，你不愿意将这么复杂的构建过程一遍又一遍的写在需要用到此对象的地方。
  3. 这个类的对象在构建过程中依赖了很多其他的类，而你无法在调用的地方提供。

UML类图

下图是简单工厂方法模式的UML图：

简单工厂模式就3个角色：

  • 对象工厂（SimpleComputerFactory）
  • 要生产的对象的接口（Computer）
  • 要生产的对象（MacComputer，MiComputer）

MacComputer 与MiComputer继承了Computer类，SimpleComputerFactory类可以根据不同的条件来生成相应品牌的电脑。

简单工厂模式的核心就是通过一个工厂方法根据不同的条件生产同一类型的产品。例如此例中我们要生产的小米笔记本和苹果笔记本他们是同一类型的产品，会实现同一个接口。

步骤：

1、定义对象抽象基类

定义一个电脑的抽象基类，包含一个为电脑安装操作系统的抽象方法，这个就是产品要实现的共同接口。

xxxxxxxxxx
public abstract class Computer {
    public abstract void setOperationSystem();
}

2、定义具体的对象类

定义具体品牌的电脑类，苹果电脑和小米电脑。

xxxxxxxxxx
public class MacComputer extends Computer {
    @Override
    public void setOperationSystem() {
        System.out.println("Mac笔记本安装Mac系统");
    }
}
public class MiComputer extends Computer {
    @Override
    public void setOperationSystem() {
        System.out.println("小米笔记本安装Win10系统");
    }
}

3、定义简单工厂方法类

使用一个静态工厂方法来根据不同的条件来产生不同的产品实例

xxxxxxxxxx
public class SimpleComputerFactory {
    public static Computer makeComputer(String brand) {
        Computer computer=null;
        switch (brand) {
            case "mac":
                computer=new MacComputer();
                break;
            case "mi":
                computer=new MiComputer();
                break;
            default:
                break;
        }
        return computer;
    }
}

从上面的代码可见，这里其实还用到了里氏替换原则（Liskov Substitution Principle），返回的对象类型是声明类型的子类，也就是面向对象的多态特性。

2.1.3 抽象工厂模式

抽象工厂模式（Abstract Factory Pattern）：提供一个抽象工厂接口，定义了一组用于创建一系列相关或依赖对象的方法，每个具体的工厂类负责创建一族产品。

首先用代码展示，再用理论解释：

代码如下：

首先，我们定义抽象产品接口 Car，其中包含一个抽象方法 drive()：

xxxxxxxxxx
// 抽象产品接口
public interface Car {
    void drive();
}

接下来，我们定义两个具体产品类 SedanCar 和 SUVCar，它们实现了抽象产品接口：

xxxxxxxxxx
// 具体产品类 SedanCar
public class SedanCar implements Car {
    @Override
    public void drive() {
        System.out.println("Driving Sedan Car");
    }
}


// 具体产品类 SUVCar
public class SUVCar implements Car {
    @Override
    public void drive() {
        System.out.println("Driving SUV Car");
    }
}

然后，我们定义抽象工厂接口 CarFactory，其中包含一个抽象方法 produceCar()：

xxxxxxxxxx
// 抽象工厂接口
public interface CarFactory {
    Car produceCar();
}

接下来，我们定义两个具体工厂类 SedanCarFactory 和 SUVCarFactory，它们实现了抽象工厂接口，并负责创建具体产品：

xxxxxxxxxx
// 具体工厂类 SedanCarFactory
public class SedanCarFactory implements CarFactory {
    @Override
    public Car produceCar() {
        return new SedanCar();
    }
}


// 具体工厂类 SUVCarFactory
public class SUVCarFactory implements CarFactory {
    @Override
    public Car produceCar() {
        return new SUVCar();
    }
}

最后，我们可以编写客户端代码来使用抽象工厂方法，创建具体的产品对象并执行操作：

xxxxxxxxxx
public class Client {
    public static void main(String[] args) {
        // 创建 SedanCar 工厂
        CarFactory sedanCarFactory = new SedanCarFactory();
        // 生产 SedanCar
        Car sedanCar = sedanCarFactory.produceCar(); // 得到了汽车产品的对象
        // 驾驶 SedanCar
        sedanCar.drive();


        // 创建 SUVCar 工厂
        CarFactory suvCarFactory = new SUVCarFactory();
        // 生产 SUVCar
        Car suvCar = suvCarFactory.produceCar();
        // 驾驶 SUVCar
        suvCar.drive();
    }
}

以上代码示例展示了如何使用抽象工厂方法模式创建不同类型的汽车（SedanCar 和 SUVCar），并通过调用抽象产品接口的方法来操作不同的汽车对象。通过使用不同的具体工厂类，我们可以创建不同类型的汽车对象，而不需要直接依赖具体的产品类。这样，客户端代码与具体产品类解耦，并提供了扩展性和灵活性。

显然，上面设计到四种类：

以下是抽象工厂方法的基本结构：

  1. 抽象产品（Abstract Product）：定义了产品的抽象接口，包括产品所具有的方法。
  2. 具体产品（Concrete Product）：实现了抽象产品接口，具体产品是抽象工厂所创建的对象。
  3. 抽象工厂（Abstract Factory）：定义了创建一组相关或依赖对象的接口。它通常是一个抽象类或接口，其中声明了一组用于创建不同产品的方法。
  4. 具体工厂（Concrete Factory）：实现了抽象工厂接口，并负责创建具体的产品对象。每个具体工厂类通常负责一组相关的产品。

抽象工厂方法的主要目的是将对象的创建与使用代码分离（在客户端实例代码中，我们没创建产品的对象，交给了具体的工厂去创建，在这里只负责使用），以实现解耦和灵活性。它通过引入一个抽象工厂接口和多个具体工厂类来实现。

抽象工厂方法的使用流程如下：

  1. 客户端代码通过与抽象工厂进行交互来创建产品对象，而不是直接实例化具体产品。(实现了客户端和具体产品的解耦)
  2. 客户端通过抽象工厂接口调用具体工厂的方法，由具体工厂创建具体产品。
  3. 客户端使用抽象产品接口来操作产品对象，而不需要关心具体产品的实现。

抽象工厂方法模式的优点包括：

  • 封装性：客户端代码与具体产品类解耦，只依赖于抽象工厂和抽象产品接口。
  • 扩展性：可以轻松添加新的具体工厂类和产品类，符合开闭原则。
  • 产品族的一致性：通过使用同一抽象工厂，可以创建一系列相关的产品，确保产品之间的兼容性和一致性。

然而，抽象工厂方法也有一些限制和注意事项：

  • 增加新的产品族较为困难：如果需要添加新的产品族，需要修改抽象工厂接口和所有的具体工厂类。
  • 增加新的产品等级结构较为困难：如果需要添加新的产品等级结构，需要修改抽象工厂接口和所有的具体工厂类。
  • 可能导致类的爆炸性增长：随着产品族和产品等级结构的增加，具体工厂类的数量可能会呈指数级增长。

抽象工厂方法适用于需要创建一组相关或依赖对象的场景，并且希望客户端代码与具体产品解耦的情况。它提供了一种灵活的方式来实现对象的创建和使用之间的解耦。

2.1.4 工厂方法模式

xxxxxxxxxx
// 抽象产品
interface Product {
    void operation();
}


// 具体产品 A
class ConcreteProductA implements Product {
    @Override
    public void operation() {
        System.out.println("具体产品 A 的操作");
    }
}


// 具体产品 B
class ConcreteProductB implements Product {
    @Override
    public void operation() {
        System.out.println("具体产品 B 的操作");
    }
}
// 抽象工厂
interface Factory {
    Product createProduct();
}


// 具体工厂 A
class ConcreteFactoryA implements Factory {
    @Override
    public Product createProduct() {
        return new ConcreteProductA();
    }
}


// 具体工厂 B
class ConcreteFactoryB implements Factory {
    @Override
    public Product createProduct() {
        return new ConcreteProductB();
    }
}
// 客户端代码
public class Client {
    public static void main(String[] args) {
        Factory factoryA = new ConcreteFactoryA();
        Product productA = factoryA.createProduct();
        productA.operation(); // 输出：具体产品 A 的操作
        
        Factory factoryB = new ConcreteFactoryB();
        Product productB = factoryB.createProduct();
        productB.operation(); // 输出：具体产品 B 的操作
    }
}

从此可以看出来，工厂方法模式和抽象工厂模式非常像。但是应用场景却不同。

在上述代码中，定义了一个抽象产品接口 Product，以及两个具体产品类 ConcreteProductA 和 ConcreteProductB，它们实现了抽象产品接口。

然后，定义了一个抽象工厂接口 Factory，其中的 createProduct 方法用于创建产品对象。

接着，定义了两个具体工厂类 ConcreteFactoryA 和 ConcreteFactoryB，它们分别实现了抽象工厂接口，并实现了对应的 createProduct 方法，用于创建具体的产品对象。

最后，在客户端代码中，通过实例化具体的工厂对象，然后调用 createProduct 方法来创建产品对象，并调用其操作方法。

工厂方法模式和抽象工厂模式对比：

工厂方法模式通过将产品的创建过程委托给具体的工厂类，实现了客户端和具体产品的解耦。客户端只需要与抽象工厂接口进行交互，无需关心具体的产品类和工厂类的实现细节。这样可以实现更灵活的对象创建方式，支持扩展新的产品类和工厂类。

工厂方法模式和抽象工厂模式在某种程度上有相似之处，因为它们都属于创建型设计模式，并且都涉及到对象的创建和实例化。

共同点：

  • 工厂方法模式和抽象工厂模式都通过引入一个抽象的工厂接口或抽象类来定义对象的创建。
  • 两种模式都支持将对象的创建过程与客户端代码解耦，提供了一种统一的方式来创建对象。
  • 工厂方法模式和抽象工厂模式都可以增强代码的可维护性和灵活性，支持客户端对产品的变化和扩展。

然而，工厂方法模式和抽象工厂模式之间存在一些关键的区别：

  1. 对象的级别和结构：
  • 工厂方法模式关注单个产品的创建，每个具体工厂类对应创建一个特定的产品。
  • 抽象工厂模式关注一组相关产品的创建，每个具体工厂类负责创建特定的产品族。
  1. 创建的灵活性：
  • 工厂方法模式将对象的创建延迟到具体工厂类中，以支持客户端对产品的变化。
  • 抽象工厂模式将对象的创建延迟到具体工厂类中，并且在不同的具体工厂类中创建一系列相关的产品。
  1. 关注点的不同：
  • 工厂方法模式注重于创建单个产品，强调产品的个体创建过程。
  • 抽象工厂模式注重于创建一组相关产品，强调产品的组合和整体创建过程。

尽管工厂方法模式和抽象工厂模式在某些方面相似，但它们的设计目的和应用场景略有不同。选择适当的模式取决于您的具体需求和设计上的考虑，以满足系统的灵活性、可扩展性和易维护性。

2.1.5 懒汉单例模式

把对象的创建延迟到真正需要使用对象的时候。且别的类里面无法创建该类的对象。

懒汉单例模式（Lazy Initialization Singleton Pattern）是一种常见的单例模式实现方式，它延迟实例化单例对象，直到第一次被使用时才进行初始化。

以下是懒汉单例模式的一种常见实现方式：

xxxxxxxxxx
public class LazySingleton {
    private static LazySingleton instance;
    private LazySingleton() {
        
    }
    public static sychronized LazySingleton getInstance() {
        if (instance == null) {
            instance = new LazySingleton();
        }
        return instance;
    }
}

在懒汉单例模式中，使用了一个静态变量 instance 来保存单例对象。在首次调用 getInstance() 方法时，会检查 instance 是否为 null，如果为 null，则创建一个新的 LazySingleton 对象并赋值给 instance。如果 instance 不为 null，则直接返回已存在的单例对象。

需要注意的是，懒汉单例模式在多线程环境下需要考虑线程安全性。上述实现中通过在 getInstance() 方法上添加 synchronized 关键字，实现了简单的线程安全性。但是由于加锁的开销较大，可能会影响性能。

如果对性能有较高的要求，可以考虑使用双重检查锁定（Double-Checked Locking）机制来提高懒汉单例模式的性能，如下所示：

xxxxxxxxxx
public class LazySingleton {
    private volatile static LazySingleton instance;
    private LazySingleton() {
        // 私有构造函数，防止外部实例化
    }
    public static LazySingleton getInstance() {
        if (instance == null) {
            sychronized(LazySingleton.class) {
                if (instance == null) {
                    instance = new LazySingleton();
                }
            }
        }
        return instance;
    }
}

在双重检查锁定机制中，通过使用 synchronized 关键字来创建同步块，只有在 instance 为 null 的情况下才会加锁。这样可以避免每次调用 getInstance() 方法都进行加锁，提高了性能。

懒汉单例模式适用于需要延迟加载单例对象的场景，避免了在应用启动时就进行对象初始化的开销。但需要注意的是，在多线程环境下，需要保证线程安全性，可以使用 synchronized 关键字或双重检查锁定机制来实现线程安全。

2.1.6 饿汉单例模式

只要类一加载，那么马上就创建该类的对象。且别的类里面无法创建该类的对象。

饿汉单例模式（Eager Initialization Singleton Pattern）是一种常见的单例模式实现方式，它在类加载时就立即创建并初始化单例对象。（类一加载就会加载类变量）

以下是饿汉单例模式的一种常见实现方式：

xxxxxxxxxx
public class EagerSingleton {
    private static final EagerSingleton instance = new EagerSingleton();
    private EagerSingleton() {
        
    }
    public static EagerSingleton getInstance() {
        return instance;
    }
}

在饿汉单例模式中，使用了一个私有的静态常量 instance 来保存单例对象，并在静态初始化器中进行初始化。由于静态初始化器在类加载时执行，因此在类加载时就会创建并初始化 instance 对象。

由于饿汉单例模式在类加载时就创建了单例对象，因此不需要考虑多线程环境下的线程安全性问题。每次调用 getInstance() 方法都会返回同一个预先创建的单例对象。

需要注意的是，饿汉单例模式具有提前实例化的特点，这意味着即使在实际使用之前，单例对象就已经被创建和初始化。这可能会导致一些不必要的资源消耗。因此，饿汉单例模式适用于单例对象较小且初始化过程较为简单的情况。

总结：

  • 饿汉单例模式在类加载时就创建并初始化单例对象。
  • 饿汉单例模式不需要考虑多线程环境下的线程安全性，因为在类加载时就创建了单例对象。
  • 饿汉单例模式适用于单例对象较小且初始化过程较为简单的情况。如果单例对象较大或初始化过程复杂，可能会造成资源浪费和启动时间延长的问题。
  • 饿汉单例模式一般使用静态常量的方式来保存单例对象，也可以使用静态代码块或静态方法的方式实现。

补充知识：

当Java程序中的类被加载时，Java虚拟机（JVM）会执行一系列步骤来完成类的加载过程。以下是类加载过程的主要步骤：

  1. 加载（Loading）：将类的字节码数据加载到内存中。这可以通过从文件系统、网络或其他来源获取字节码数据来完成。加载后的字节码数据存储在方法区（JDK 7及更早版本）或元空间（JDK 8及以后版本）中。
  2. 验证（Verification）：对字节码进行验证，确保其符合Java虚拟机规范，并且不会对虚拟机本身造成安全威胁。验证过程包括静态验证（验证类的结构和元数据）、字节码验证（验证字节码的合法性和安全性）和符号引用验证（验证类的引用是否正确）。
  3. 准备（Preparation）：为类的静态变量分配内存，并设置初始值。这些静态变量包括类变量和常量。非静态变量则会在对象实例化时进行准备和初始化。
  4. 解析（Resolution）：将类、接口、字段和方法的符号引用解析为直接引用。符号引用是在编译时生成的，直接引用是指向内存地址的指针或句柄。解析过程可以是静态的（编译期间解析）或动态的（运行期间解析）。
  5. 初始化（Initialization）：对类进行初始化操作。这是类加载过程的最后一步。在初始化阶段，会执行类构造器（即 <clinit> 方法）的代码，包括对静态变量赋值和执行静态代码块中的代码。类构造器是由编译器自动生成的，用于初始化类的静态成员。

需要注意的是，类加载过程是按需进行的，即只有在首次使用该类时才会进行加载。此外，类加载过程具有层次性，即当一个类加载时，它所依赖的其他类也会被加载。类加载器（ClassLoader）负责执行类加载过程，根据特定的加载策略加载类。

类加载过程的细节可能因JVM的实现而有所不同，上述描述是一般情况下的类加载过程。类加载过程是Java语言实现跨平台性和动态性的重要基础。

2.2 行为型模式

2.2.1 模板方法模式

模板方法模式（Template Method Pattern）是一种行为型设计模式，它定义了一个算法的框架，将一些步骤的实现延迟到子类中，以便子类可以重新定义算法的某些步骤而不改变算法的整体结构。

在模板方法模式中，定义一个抽象类或接口作为模板，其中包含一个模板方法和一些抽象方法或钩子方法。模板方法定义了算法的骨架，通过调用抽象方法或钩子方法来完成算法中的具体步骤，而具体的实现则由子类提供。

在上述示意图中，AbstractClass 是抽象类或接口，其中包含一个模板方法 templateMethod()，以及一些抽象方法或钩子方法，如 abstractMethod1()、abstractMethod2() 等。具体的子类（如 ConcreteClass1、ConcreteClass2）继承抽象类或实现接口，并提供这些抽象方法的具体实现。

下面是示例代码：

xxxxxxxxxx
abstract class AbstractClass {
    public void templateMethod() {
        step1();
        step2();
        step3();
    }


    public abstract void step1();


    public abstract void step2();


    public abstract void step3();
}


class ConcreteClass1 extends AbstractClass {
    @Override
    public void step1() {
        System.out.println("ConcreteClass1: Step 1");
    }


    @Override
    public void step2() {
        System.out.println("ConcreteClass1: Step 2");
    }


    @Override
    public void step3() {
        System.out.println("ConcreteClass1: Step 3");
    }
}


class ConcreteClass2 extends AbstractClass {
    @Override
    public void step1() {
        System.out.println("ConcreteClass2: Step 1");
    }


    @Override
    public void step2() {
        System.out.println("ConcreteClass2: Step 2");
    }


    @Override
    public void step3() {
        System.out.println("ConcreteClass2: Step 3");
    }
}


public class Client {
    public static void main(String[] args) {
        AbstractClass class1 = new ConcreteClass1();
        class1.templateMethod();


        System.out.println("----------------------");


        AbstractClass class2 = new ConcreteClass2();
        class2.templateMethod();
    }
}

在上述示例中，我们定义了一个抽象类 AbstractClass，其中包含了一个模板方法 templateMethod() 和三个抽象方法 step1()、step2() 和 step3()。具体的子类 ConcreteClass1 和 ConcreteClass2 继承了 AbstractClass 并实现了抽象方法。

客户端代码通过实例化具体子类对象并调用 templateMethod() 方法来触发模板方法的执行。在模板方法中，按照定义的算法骨架，依次调用了抽象方法，而具体子类则提供了这些抽象方法的具体实现。

运行客户端代码，将会输出以下结果：

xxxxxxxxxx

ConcreteClass1: Step 2
ConcreteClass1: Step 3
----------------------

ConcreteClass2: Step 3

可以看到，模板方法模式确保了算法的一致性，同时允许具体子类根据需要重新定义算法的某些步骤，而不影响整体结构。
